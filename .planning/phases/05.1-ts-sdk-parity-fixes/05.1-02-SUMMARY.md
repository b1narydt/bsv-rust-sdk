# Phase 05.1 Plan 02: BRC-29 Config Extension and Validation Parity Summary

**One-liner:** BRC-29 config extended with refund fields and InternalizeProtocol enum; ensure_valid_option now validates protocol_id, labels, and description to match TS SDK exactly.

## What Was Built

**Task 1: Config extension and InternalizeProtocol enum (PARITY-08)**

Added `InternalizeProtocol` enum to `brc29.rs` with `WalletPayment` ("wallet payment") and `BasketInsertion` ("basket insertion") variants. Extended `Brc29RemittanceModuleConfig` with three new fields matching TS SDK constructor defaults: `refund_fee_satoshis: 1000`, `min_refund_satoshis: 1000`, `internalize_protocol: InternalizeProtocol::WalletPayment`. Updated `Default` impl accordingly.

**Task 2: Validation extensions and is_atomic_beef docs (PARITY-09, PARITY-11)**

Extended `ensure_valid_option` with three new validation checks mirroring TS SDK lines 304-324: protocol_id protocol string non-empty, labels all non-empty/non-whitespace, description non-empty/non-whitespace. Added doc comments to `ensure_valid_settlement` noting u32 type guarantees outputIndex >= 0. Replaced `is_atomic_beef` doc with accurate byte-level semantics description matching TS SDK behavior.
